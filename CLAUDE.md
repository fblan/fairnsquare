# Project: SplitNShare (FairNSquare)

## Implementation Documentation

After completing any feature, bug fix, or refactor implementation, generate a documentation file:

- **Location**: `src/main/doc/implementation/`
- **Naming**: `YYYY-MM-DD-Feature-<name-of-the-feature>.md` (or `Bugfix-` / `Refactor-` as appropriate)
- **Structure**:
  1. **What, Why and Constraints** — Describe what was done, the motivation, and any technical constraints followed
  2. **How** — Explain the approach taken step by step (files created, modified, deleted and why)
  3. **Tests** — List automated tests added, or note manual testing performed

## Development Rules

Follow the rules defined in `src/doc/rules/`:
- `frontend-rules.md` — Frontend testing and development rules
- `backend-rules.md` — Backend development rules

## Continuous Improvement

After completing any implementation that modifies the frontend, review `src/doc/rules/frontend-rules.md` and propose improvements or new rules based on patterns, issues, or best practices observed during the implementation. Present proposals to the user for approval before modifying the rules file.
After completing any implementation that modifies the backend, review `src/doc/rules/backend-rules.md` and propose improvements or new rules based on patterns, issues, or best practices observed during the implementation. Present proposals to the user for approval before modifying the rules file.
